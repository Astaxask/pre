# Data Schema — Personal Reality Engine

> This is the source of truth for all data structures in the PRE.
> The TypeScript types in `packages/shared/src/life-schema.ts` must match this document exactly.
> If you change this document, update the types. If you change the types, update this document.

---

## Core principle: everything is a LifeEvent

Every piece of data that enters the system — a bank transaction, a sleep reading, a calendar event, a conversation message — gets normalized into a `LifeEvent`. This single unified shape is what makes cross-domain reasoning possible. The inference engine only knows about `LifeEvent` objects; it never touches raw API responses.

---

## LifeEvent

The root type. Every record in the system is a `LifeEvent`.

```typescript
type LifeEvent = {
  // Identity
  id: string                    // UUID v4, generated on ingestion
  source: DataSource            // Where this came from (e.g. 'plaid', 'healthkit')
  sourceId: string              // Original ID from the source system (for deduplication)
  
  // Classification
  domain: LifeDomain            // Which life domain this belongs to
  eventType: string             // Fine-grained type within domain (e.g. 'transaction.expense')
  
  // Time
  timestamp: number             // Unix ms — when the event happened in real life
  ingestedAt: number            // Unix ms — when we received it
  
  // Content
  payload: DomainPayload        // Typed per domain — see below
  
  // Memory
  embedding: number[] | null    // 1536-dim vector, null until embedded
  summary: string | null        // 1-2 sentence LLM summary for RAG, null until generated
  
  // Privacy & trust
  privacyLevel: PrivacyLevel    // Controls whether this can leave the device
  confidence: number            // 0–1, how reliable is this data point
}
```

## Supporting types

```typescript
type LifeDomain = 
  | 'body'       // Health, sleep, fitness, biometrics
  | 'money'      // Transactions, balances, net worth, bills
  | 'people'     // Relationships, communications, social signals
  | 'time'       // Calendar events, time blocks, commitments
  | 'mind'       // Learning, goals, mood, cognition
  | 'world'      // External events, news, weather, location context

type PrivacyLevel = 
  | 'private'       // Never leaves device. Raw health readings, transactions, messages.
  | 'summarizable'  // Can be abstracted into a cloud-safe summary, but raw form stays local.
  | 'cloud-safe'    // Abstract enough to send to cloud models. No PII, no raw numbers.

type DataSource =
  | 'plaid'
  | 'healthkit'
  | 'google-fit'
  | 'oura'
  | 'whoop'
  | 'garmin'
  | 'google-calendar'
  | 'gmail'
  | 'manual'       // User entered directly
  | 'inferred'     // Generated by the inference engine, not from an external source
```

## Domain payloads

Each domain has a strongly-typed payload. The `payload` field on `LifeEvent` is a discriminated union of these.

### Body

```typescript
type BodyPayload = {
  domain: 'body'
  subtype: 
    | 'sleep'
    | 'hrv'               // Heart rate variability
    | 'resting-hr'
    | 'activity'          // Steps, active calories, workout
    | 'recovery-score'    // Oura/WHOOP readiness
    | 'biometric'         // Weight, blood pressure, glucose, etc.
    | 'symptom'           // User-logged symptom

  // Sleep
  sleepDuration?: number        // minutes
  sleepScore?: number           // 0–100 if source provides it
  deepSleepMinutes?: number
  remSleepMinutes?: number
  
  // HRV / HR
  hrvMs?: number                // RMSSD in milliseconds
  restingHeartRate?: number     // bpm
  
  // Activity
  steps?: number
  activeCalories?: number
  workoutType?: string          // 'run' | 'strength' | 'cycling' | etc.
  workoutDurationMinutes?: number
  
  // Recovery
  recoveryScore?: number        // 0–100
  strainScore?: number          // 0–21 for WHOOP
  
  // Biometric
  metric?: string               // 'weight' | 'blood_pressure_systolic' | etc.
  value?: number
  unit?: string                 // 'kg' | 'mmHg' | 'mg/dL' | etc.
  
  // Symptom
  symptomDescription?: string
  severity?: number             // 1–5
}
```

### People

```typescript
type PeoplePayload = {
  domain: 'people'
  subtype:
    | 'communication'         // Email sent/received, message
    | 'meeting'               // Call, video, in-person
    | 'relationship-signal'   // Inferred: relationship health indicator

  // Communication
  channel?: 'email' | 'sms' | 'slack' | 'whatsapp' | 'other'
  direction?: 'sent' | 'received'
  contactId?: string          // Opaque local ID — never store actual email/phone here
  
  // NOTE: message content is NEVER stored in the payload.
  // Only metadata: who, when, channel, direction.
  // privacyLevel on these events must always be 'private'.
  
  // Relationship signal (inferred by engine, not from raw comms)
  contactId?: string
  signalType?: 'frequency-drop' | 'frequency-increase' | 'long-silence' | 'reconnect'
  daysSinceLastContact?: number
}
```

### Mind

```typescript
type MindPayload = {
  domain: 'mind'
  subtype:
    | 'goal'
    | 'goal-progress'
    | 'mood-log'              // User-logged
    | 'learning-session'      // Reading, course, practice
    | 'reflection'            // User-written note or journal entry

  // Goal

  goalTitle?: string

  targetDate?: number         // Unix ms

  // Goal progress
  goalId?: string
  progressPercent?: number
  progressNote?: string
  
  // Mood
  valence?: number            // -1.0 to 1.0 (negative to positive)
  arousal?: number            // 0.0 to 1.0 (calm to activated)

  topic?: string

  medium?: 'book' | 'article' | 'course' | 'video' | 'practice' | 'other'
```

### World

```typescript
type WorldPayload = {
  domain: 'world'
  subtype:
    | 'weather'
    | 'location-context'      // Coarse: 'home' | 'work' | 'traveling' — never GPS coords

  eventCategory?: 'economic' | 'health' | 'weather-extreme' | 'local' | 'other'

}
```

## SQLite tables

These map 1:1 to the schema above. Drizzle handles migrations.

```sql
-- Main event store
CREATE TABLE life_events (
  id           TEXT PRIMARY KEY,
  source       TEXT NOT NULL,
  source_id    TEXT NOT NULL,
  domain       TEXT NOT NULL,
  event_type   TEXT NOT NULL,
  timestamp    INTEGER NOT NULL,
  ingested_at  INTEGER NOT NULL,
  payload      TEXT NOT NULL,    -- JSON blob, typed by domain
  summary      TEXT,             -- LLM-generated, nullable
  privacy_level TEXT NOT NULL DEFAULT 'private',
  confidence   REAL NOT NULL DEFAULT 1.0,
  UNIQUE(source, source_id)      -- Prevent duplicate ingestion
);

CREATE INDEX idx_events_domain    ON life_events(domain);
CREATE INDEX idx_events_timestamp ON life_events(timestamp);
CREATE INDEX idx_events_source    ON life_events(source);

-- Embeddings live separately (LanceDB manages them, but this table tracks sync status)
CREATE TABLE embedding_sync (
  event_id     TEXT PRIMARY KEY REFERENCES life_events(id),
  embedded_at  INTEGER,
  model        TEXT              -- Which embedding model was used
);

-- Goals (Mind domain goals get a dedicated table for fast access)
CREATE TABLE goals (
  id           TEXT PRIMARY KEY,
  title        TEXT NOT NULL,
  domain       TEXT NOT NULL,
  target_date  INTEGER,
  status       TEXT NOT NULL DEFAULT 'active',
  created_at   INTEGER NOT NULL,
  updated_at   INTEGER NOT NULL
);

-- Trigger rule firing log (prevents cooldown violations)
CREATE TABLE trigger_log (
  id           TEXT PRIMARY KEY,
  rule_id      TEXT NOT NULL,
  fired_at     INTEGER NOT NULL,
  severity     TEXT NOT NULL,
  dismissed_at INTEGER            -- null = user hasn't seen/dismissed yet
);

-- Integration sync state (tracks last successful sync per source)
CREATE TABLE integration_sync (
  source       TEXT PRIMARY KEY,
  last_sync_at INTEGER,
  cursor       TEXT,              -- Source-specific pagination cursor
  status       TEXT NOT NULL DEFAULT 'idle'
);
```

## Encryption at rest

Everything in SQLite is encrypted using `better-sqlite3-sqlcipher` with an AES-256 key derived from the `PRE_ENCRYPTION_KEY` env var. The key never touches disk in plaintext.

The `payload` column for events with `privacy_level = 'private'` gets an additional layer of field-level encryption using `libsodium` secretbox before being written to the JSON blob. This means even if someone extracts the SQLite key, private payloads remain protected.

LanceDB vectors are stored in an encrypted directory using `age` with the same derived key.

---

## What the inference engine sees

The inference engine does NOT query `life_events` directly. It calls the sidecar which returns normalized `LifeEvent` objects after decryption. The engine operates on these fields only:

- `domain`, `eventType`, `timestamp`
- `payload` (decrypted, typed)
- `embedding` (from LanceDB, via similarity search)
- `summary` (for RAG context)

The engine never sees `sourceId` and never constructs queries that could re-identify a specific account, person, or location beyond what the user explicitly gave it.
